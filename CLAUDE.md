# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

> **Project conventions live in [`PROJECT_RULES.md`](./PROJECT_RULES.md).**
> Read it before changing API clients, auth flows, env vars, error handling,
> or build/deploy config. This file covers what `PROJECT_RULES.md` does not:
> the architecture map, repo layout, legacy hazards, and how to work with
> Claude in this repo.

## Architecture

- **Mobile/Web frontend** — Expo SDK 52, React Native 0.76, React 18.
  - Routing: Expo Router (file-based) under `app/`.
  - State: local React state + AsyncStorage; Supabase for auth + user data.
  - Web shell: Next.js (`next.config.mjs`) for landing/preview pages;
    Expo web (`react-native-web`) for the app itself.
  - Distribution: EAS Build/Submit for iOS + Android; Vercel/Fly for web.
- **Backend** (`backend/`) — FastAPI on Python 3.11.
  - ORM: tortoise-orm + aerich; aiosqlite locally, Supabase Postgres in prod.
  - NLP: NLTK VADER, TextBlob, FinBERT (torch).
  - AI: Groq (chat/summarization), HuggingFace.
  - Data sources: Alpha Vantage, Kalshi prediction markets, RSS/news
    aggregation (feedparser, aiohttp, BeautifulSoup).
  - Push notifications: exponent-server-sdk → Expo push tokens.
  - Deploy targets: Google Cloud Run (primary), plus Dockerfiles for
    Fly.io, Railway, Render.

## Repo layout (canonical)

- `app/` — Expo Router screens, components, services, hooks, providers.
- `backend/` — FastAPI app, NLP/AI services, DB layer, news pipeline.
- `assets/`, `app-store-assets/` — image and store assets.
- `__tests__/`, `backend/tests/` — JS/TS and Python test suites.
- `scripts/` — build, icon generation, automation.
- `expo-plugins/` — custom Expo config plugins.
- `supabase/` — SQL migrations, edge function config, setup SQL.

## Common commands

- `npm run dev` — start Expo dev server.
- `npm run ios` / `npm run android` — native run via Expo.
- `npm test` — Jest suite. Single test: `npx jest path/to/file.test.tsx`
  or `npx jest -t "test name"`.
- `npm run lint` / `npm run lint:fix` — ESLint.
- `npm run build:ios` / `build:android` — EAS production build.
- `npm run submit:ios` / `submit:android` — EAS Store submit.
- Backend dev: `cd backend && uvicorn main:app --reload --port 8000`.
- Backend tests: `cd backend && pytest`. Single test:
  `pytest path/to/test_file.py::test_name`.

## Legacy hazards — read before editing

- **Top-level `.tsx`/`.js` files** (`MainApp.js`, `App.web.js`,
  `App.web.new.js`, `ai-analysis-overlay.tsx`, `app-icon*.tsx`, etc.)
  are legacy duplicates of files now under `app/`. Edit the `app/`
  versions, not these.
- **Three lockfiles tracked** (`package-lock.json`, `pnpm-lock.yaml`,
  `yarn.lock`). npm is the intended manager; the other two are stale.
  Don't regenerate them — if you change deps, update `package-lock.json`
  only.
- **Multiple backend entrypoints — `main_simple_nlp.py` is canonical**
  (verified empirically: 25 routes including all `/api/sentiment`,
  `/api/lexicon`, `/api/news/*`, `/api/dashboard/sentiment-engine`,
  `/api/prediction-market/*`). `main.py` is a thin stub with only 8
  routes (`/`, `/health`, `/analyze-sentiment`, 5 `/notifications/*`) —
  it does NOT load most production endpoints. `main_integrated.py` and
  `main_production.py` are older alternate entries; only touch them if
  you understand why they exist. Production deployments should run
  `uvicorn main_simple_nlp:app`, not `main:app`.
- **`AIAnalysisOverlay` is currently disabled** (commit `85d01aea`) as
  part of the Build 26 stability rollback. Don't re-enable it without
  also addressing the incompatible dependencies that caused the disable.
- **`ios/` is intentionally minimal** — only `IntegraMarkets.xcworkspace`
  is tracked; the rest is regenerated by EAS managed prebuild. Don't
  add native iOS source files here; configure via `app.json` and
  `expo-plugins/` instead.
- **`expo-plugins/disable-privacy-aggregation.cjs`** exists to satisfy
  App Store privacy review — don't remove without a replacement plan.
- **Multiple deploy configs coexist** (`Dockerfile`, `Dockerfile.web`,
  `Dockerfile.test`, `fly.toml`, `fly.web.toml`, `railway.toml`,
  `vercel.json`). Cloud Run is primary for backend; check before
  modifying any of these.
- **iOS build requires `useFrameworks: "static"` in `expo-build-properties`** —
  `@react-native-google-signin/google-signin@15+` pulls in `AppCheckCore`
  which needs modular headers. Removing the plugin block or dropping the
  `static` value will cause `pod install` to fail on EAS Build with:
  > The Swift pod `AppCheckCore` depends upon `GoogleUtilities` and
  > `RecaptchaInterop`, which do not define modules.
  Fix landed 2026-07-01 (build `e6090704-5e31-41ae-bab7-b0c89f606008`).
- **iOS build requires `patches/react-native+0.76.9.patch`** — RN 0.76 pulls
  fmt v11.0.2, whose `basic_format_string` constructor is `consteval`. Xcode 26
  rejects this at compile time. The patch downgrades fmt to v9.1.0 in
  `third-party-podspecs/fmt.podspec` (permissive `constexpr` constructor).
  Requires `patch-package` in devDeps + `"postinstall": "patch-package"` in
  package.json scripts. Delete/skip either and every Xcode 26 build will fail
  with `call to consteval function 'fmt::basic_format_string<...>'`.
  First working Xcode 26 build: `96779f72-0d89-47aa-ae84-20a4936a2206`
  (build 62, 2026-07-02). Revert once RN 0.77+ / Expo SDK 53 is adopted.
- **Never use plain `npm install` for packages with native code** — always
  `npx expo install <pkg>` so the version matches the SDK. Native packages
  installed with wrong versions cause silent CocoaPods failures on EAS.
  Known offenders installed too-new during 2026-07-01 session:
  `@react-native-community/netinfo` (needs 11.4.1 for SDK 52, not latest),
  `expo-clipboard` (needs ~7.0.1, not 8.x).

## Working with Claude

- When unsure which `App.*.js` / `main_*.py` variant is canonical, ask
  before editing.
- For UI changes, run the Expo dev server and exercise the feature in
  iOS simulator before reporting done; lint/tests alone are not enough.
- For backend changes, run `pytest` and start uvicorn locally to
  confirm imports resolve before claiming success.
- Prefer fixing root causes over flags/fallbacks. The repo already has
  parallel implementations of several features — don't add another.
- All conventions in `PROJECT_RULES.md` are load-bearing for the build
  process. If a rule there seems wrong, flag it rather than silently
  diverging.

## Code Review Standards

After completing any implementation, run `/simplify` before presenting code.

Flag:
- Functions longer than 30 lines
- Logic duplicated more than twice (extract to utility)
- Any `any` type usage in TypeScript
- Components with more than 3 props that could be grouped into an object
- Missing error handling on async operations
